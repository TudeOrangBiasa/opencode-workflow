# Skill Placement Boundaries

## Boundary

Skills are organized by how often they should be active, not by how interesting they are.

## Buckets

- `skills/engineering/` — daily engineering workflows.
- `skills/productivity/` — daily non-code workflow helpers.
- `skills/misc/` — occasional specialist skills, loaded only when the domain appears.
- `skills/personal/` — personal setup, not promoted or linked by default.
- `skills/in-progress/` — drafts not ready to ship.
- `skills/deprecated/` — inactive archive.

## Out of Scope for Engineering Bucket

Do not put occasional specialist checklists in `engineering/`:

- PHP/Laravel review
- database review
- security review
- Docker patterns
- React patterns/performance
- shared hosting deployment
- team handoff/version/changelog
- verification evidence checklist

These belong in `misc/` unless they become daily workflow primitives.

## Trigger Discipline

No broad “always useful” triggers. Misc skills must say “Use only when...” or otherwise narrow activation.

If a skill starts triggering too often, tighten description before deleting it.
